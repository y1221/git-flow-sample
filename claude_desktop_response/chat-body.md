## 現状の対応表

まず、ワークフローのトリガーと、各ブランチ保護ルールで要求しているステータスチェックを突き合わせます。

## 検出された問題

### 問題1：`main`の保護ルールに `check-commit-message` が不足している

`main`へのPRは `release/*` または `hotfix/*` から来ます。ワークフロー①はこれらのブランチへのpushで実行されるため、PRのHEADコミットにはステータスが付与されます。つまり必須チェックとして追加可能であり、追加すべきです。

現状だと、release/hotfixブランチ上のコミットメッセージ形式が不正でも、mainへのマージがブロックされません。

**対処**: `main`の `Require status checks` に `check-commit-message` を追加する。

### 問題2：`develop`の保護ルールに `check-branch-commit-rule` を追加できない（構造的制約）

`develop`への PR は `feature/*`、`release/*`、`hotfix/*` の3種類から来ます。一方、ワークフロー②は `release/**` と `hotfix/**` へのpushでしか実行されません。

もし `develop` の必須チェックに `check-branch-commit-rule` を追加すると、`feature/*` からのPRではこのステータスが一切生成されないため、PRが永久にマージ不可になります。

そのため現状の「`develop`には含めない」という判断自体は妥当です。ただし、これは `release/*` → `develop` へのPRで `feat:` コミット混入チェックが強制されないことを意味します（`main` 側での必須化で間接的にはカバーされますが、developへの先行マージ時点では防げません）。

**対処案（任意）**: もし `develop` 側でも強制したい場合は、ワークフロー②を `pull_request` トリガーに変更し、headブランチが `feature/*` のときはスキップするロジックを加えるか、別ワークフローとして分離する方法があります。

### 問題3（付随）：ワークフロー②の比較ベースが不適切

これはトリガーと保護ルールの整合性とは別の話ですが、実運用で確実に壊れる箇所なので併せて指摘します。

ワークフロー②の `git log origin/main..HEAD` という比較ベースは、`hotfix/*`（mainから分岐）には正しく機能しますが、`release/*`（developから分岐）には機能しません。

`release/*` ブランチで `origin/main..HEAD` を展開すると、developの履歴に含まれるfeatureマージ由来の `feat:` コミットがすべて含まれるため、**正常なreleaseブランチでも必ず偽陽性で失敗します**。

```bash
# 現状（release/*で誤検出する）
git log origin/main..HEAD --pretty=format:"%s" | grep -qE "^feat:"
```

```bash
# 修正案：ブランチ種別に応じて比較ベースを切り替える
BRANCH=${GITHUB_REF#refs/heads/}
if echo "$BRANCH" | grep -qE "^release/"; then
  BASE="origin/develop"
elif echo "$BRANCH" | grep -qE "^hotfix/"; then
  BASE="origin/main"
fi
git log $BASE..HEAD --pretty=format:"%s" | grep -qE "^feat:"
```

同様に、ワークフロー①も `origin/develop..HEAD` を固定で使っていますが、`hotfix/*` はmainから分岐するため、`origin/main..HEAD` にすべきです。同じ方式でブランチ種別ごとに分岐させるのが適切です。

## ステータスチェックの仕組み

`Require status checks to pass before merging` が見ているのは、**PRのHEADコミット（＝ソースブランチの最新コミット）に付与されたステータス**です。ワークフローのトリガーが `push` か `pull_request` かは関係なく、そのコミットに対してチェックが実行されてステータスが記録されていれば、条件を満たします。

具体例で見ると、`check-branch-commit-rule`（トリガー: `release/**`への`push`）の場合は次のような流れになります。

1. `release/1.0.0` ブランチにコミットをpushする
2. pushトリガーでワークフローが実行され、そのコミットにステータスが付く
3. `release/1.0.0` → `main` へのPRを作成（またはすでに作成済み）
4. PRのHEADコミットにステータスが付いているので、保護ルールの条件を満たす

つまり`push`トリガーのワークフローでも、PRのHEADコミットで実行されていれば必須チェックとして機能します。

## ただし `push` トリガーには落とし穴がある

問題になるのは **「PRを作成した後、ソースブランチに新しいpushがない場合」** です。

たとえば次のような手順を踏んだとき、チェックが実行されません。

1. `release/1.0.0` にコミットをpushする
2. pushトリガーのワークフローは**まだ必須チェックとして登録されていなかった**
3. その後、保護ルールに当該チェックを追加する
4. PRを作成する
5. 新たなpushがないのでワークフローは再実行されない → ステータスが存在しない → PRがマージ不可

また、もっと一般的なケースとして次のような問題もあります。

1. ローカルでコミットを積んでpushする → ワークフロー実行 → pass
2. GitHub UI上でPRのタイトルだけ修正する
3. 新たなpushはないので `check-pr-title` は（`pull_request` トリガーなので）再実行されるが、`push`トリガーのワークフローは再実行されない

この点、`pull_request`トリガーであればPRの作成・更新・同期のたびに確実に実行されるため、保護ルールとの相性は本質的に良いです。

## 実務上の判断

`push`トリガーのワークフローを必須チェックに設定すること自体は有効ですが、`pull_request`トリガーに比べて「ステータスが存在しない」状態が起こりやすいという弱点があります。

今回のワークフロー構成で言えば、②`check-branch-commit-rule`はブランチ上のコミット内容を検証するものなので、`push`トリガーに合理性があります。ただ、必須チェックとしてより確実に機能させたいなら、`pull_request`トリガーも併記するのが堅実です。

```yaml
on:
  push:
    branches:
      - "release/**"
      - "hotfix/**"
  pull_request:
    branches:
      - main
      - develop
```

こうすればpush時の即時フィードバックと、PR上での必須チェックとしての確実な実行を両立できます。

## 変更点のまとめ

### ワークフロー側

全ワークフローのトリガーを `pull_request` に統一しました。保護ルールの必須チェックとして確実に機能させるためです。pushトリガーでも動作はしますが、「PRのHEADコミットにステータスが付いている必要がある」という間接的な依存関係に頼ることになり、技術的強制としての明快さに欠けます。

比較ベースについては、`origin/develop` や `origin/main` を固定で参照する方式から、`github.event.pull_request.base.sha`（PRのベースコミットSHA）を使う方式に変更しました。これによりブランチの分岐元を意識する必要がなくなり、どのブランチからのPRでも正しく差分を取得できます。

ワークフロー②のトリガー対象は `main` のみに絞りました。このチェックが防ぎたいのは「release/hotfixブランチにfeat:コミットが混入した状態で本番に出ること」であり、技術的にブロックすべきポイントはmainへのマージ時点です。developへのマージは後続の統合であり、ここで同じチェックを重複させる必要はありません。

ワークフロー④のトリガーから `types: [edited]` を除外しました。このチェックはHEADブランチとBASEブランチの組み合わせを検証するものであり、PRの説明文やタイトルの編集では検査対象が変わらないためです。`edited` が必要なのはPRタイトルを検証するワークフロー③のみです。

### ブランチ保護ルール側

`main` に `check-commit-message` を追加しました。現行ではこれが欠落しており、release/hotfixブランチのコミットメッセージ形式違反がmainへのマージ時にブロックされない状態でした。

`develop` から `check-branch-commit-rule` を除外することを明示しました（現行でも設定されていませんが、意図的な判断であることを明確にするためです）。
