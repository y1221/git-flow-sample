  ---
  git-flow ブランチモデル

  Vincent Driessenが2010年に提唱したブランチ戦略です。

  メインブランチ（常に存在）

  ┌─────────────────┬─────────────────────────────────────────────────┐
  │    ブランチ     │                      役割                       │
  ├─────────────────┼─────────────────────────────────────────────────┤
  │ main (旧master) │ 本番リリース済みコードのみ。コミット=新リリース │
  ├─────────────────┼─────────────────────────────────────────────────┤
  │ develop         │ 次リリースに向けた統合ブランチ。開発の中心      │
  └─────────────────┴─────────────────────────────────────────────────┘

  サポートブランチ（必要時に作成・削除）

  feature/*

  - 分岐元: develop
  - マージ先: develop
  - 用途: 新機能開発
  - ポイント: --no-ff でマージしてフィーチャーの履歴を保持

  release/*

  - 分岐元: develop
  - マージ先: develop と main
  - 用途: リリース準備（バージョン番号確定、軽微バグ修正）
  - ポイント: 大型新機能は受け付けない

  hotfix/*

  - 分岐元: main
  - マージ先: develop と main
  - 用途: 本番環境の緊急バグ修正
  - ポイント: 開発作業を中断させずに並行対応可能

  重要なマージ戦略

  --no-ff フラグの使用を強調しています。Fast-forwardマージを避けてマージコミット
  を明示的に作成することで、フィーチャーの履歴を追跡しやすくなります。

  著者注記（2020年追記）

  継続的デリバリーを行うチームには、よりシンプルな GitHub Flow
  の採用を推奨しています。git-flowは「リリースサイクルが明確に存在するソフトウェ
  ア」に適したモデルです。

  構築したコミットツリーの概要

  網羅したgit-flowのルール

  ┌─────────────────┬────────────────────────────────────────────────────────┐
  │     ルール      │                        実施内容                        │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ mainブランチ    │ リリース済みコードのみ管理。3回のマージ（v1.0.0,       │
  │                 │ v1.0.1, v1.1.0）                                       │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ developブランチ │ 開発の統合ブランチ。全featureの統合先                  │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ feature/*       │ developから分岐・developへ--no-ffマージ後削除（login,  │
  │                 │ dashboard, profile, settings）                         │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ release/*       │ developから分岐 → バグ修正・バージョン設定 →           │
  │                 │ mainとdevelop両方へマージ後削除（1.0.0, 1.1.0）        │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ hotfix/*        │ mainから分岐 →                                         │
  │                 │ mainとdevelop両方へマージ後削除（1.0.1）               │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ --no-ffマージ   │ 全マージにFast-forwardを禁止してマージコミットを明示   │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ タグ            │ mainへのマージごとにアノテーションタグ（v1.0.0,        │
  │                 │ v1.0.1, v1.1.0）を付与                                 │
  └─────────────────┴────────────────────────────────────────────────────────┘

  ポイント

  - hotfixはdevelopの開発作業（profile, settings）と並行してmainから分岐し対応 →
   その後developにもフィードバック
  - releaseブランチは大型機能を受け付けず、バグ修正のみ実施
  - 全サポートブランチはマージ後に削除済み

  ---
  GitHubのIssueとgit-flowを連携する方法

  1. ブランチ名にIssue番号を含める

  最も広く使われているアプローチです。

  # Issue #42「ログイン機能の実装」に対応する場合
  git checkout -b feature/42-login

  # または
  git checkout -b feature/42_login
  git checkout -b feature/#42-login

  ブランチ一覧を見るだけでどのIssueに対応した作業かが分かります。

  ---
  2. コミットメッセージにIssue番号を含める

  # 言及のみ（Issueにコミットへの参照が自動でつく）
  git commit -m "feat: ログイン機能を追加 #42"

  # PRマージ時にIssueを自動クローズするキーワード
  git commit -m "feat: ログイン機能を追加 closes #42"
  git commit -m "feat: ログイン機能を追加 fixes #42"
  git commit -m "feat: ログイン機能を追加 resolves #42"

  closes / fixes / resolves のいずれかを使うと、PRがmainにマージされた時点でIssu
  eが自動的にクローズされます。

  ---
  3. Pull RequestとIssueを紐付ける

  PRの本文に同様のキーワードを書くことで紐付けます。

  ## 概要
  ログイン機能を実装しました。

  closes #42

  GitHub上でPRとIssueがリンク表示され、マージと同時にIssueがクローズされます。

  ---
  git-flowとの組み合わせイメージ

  Issue #42「ログイン機能の実装」
      ↓ feature作成
  feature/42-login
      ↓ コミット
  feat: ログイン機能を追加 #42
  feat: セッション管理を追加 #42
      ↓ developへPRを出す（PR本文に closes #42）
  develop ← Merge branch 'feature/42-login'
      ↓ PRマージ
  Issue #42 が自動クローズ

● GitHubのブランチ保護ルール（git-flow向け）

  GitHubでは Settings → Branches → Branch protection rules
  から設定します。git-flowで保護すべきブランチは main と develop の2つです。

  ---
  main ブランチ（最も厳格に保護）

  main は本番コードのみを管理するため、直接pushを完全に禁止し、必ずレビューを通
  す構成にします。

  設定項目: Require a pull request before merging
  推奨値: ✅ 有効
  理由: 直接pushを禁止。必ずPR経由にする
  ────────────────────────────────────────
  設定項目: └ Required approvals
  推奨値: 1 以上
  理由: レビューなしのマージを防ぐ
  ────────────────────────────────────────
  設定項目: └ Dismiss stale pull request approvals when new commits are pushed
  推奨値: ✅ 有効
  理由: 承認後に差し込まれた変更を見落とさない
  ────────────────────────────────────────
  設定項目: Require status checks to pass before merging
  推奨値: ✅ 有効
  理由: CIが通ったコードのみマージ可能にする
  ────────────────────────────────────────
  設定項目: └ Require branches to be up to date before merging
  推奨値: ✅ 有効
  理由: マージ前にdevelopとの差分を解消させる
  ────────────────────────────────────────
  設定項目: Require linear history
  推奨値: ❌ 無効
  理由: git-flowは --no-ff でマージコミットを作るため、有効にすると矛盾する
  ────────────────────────────────────────
  設定項目: Do not allow bypassing the above settings
  推奨値: ✅ 有効
  理由: 管理者も例外なくルールに従う
  ────────────────────────────────────────
  設定項目: Restrict who can push to matching branches
  推奨値: ✅ 有効
  理由: releaseブランチ・hotfixブランチ担当者のみに限定

  ---
  develop ブランチ（中程度の保護）

  develop
  はfeatureブランチの統合先です。mainよりは緩めつつ、直接pushは禁止します。

  ┌──────────────────────────┬────────┬─────────────────────────────────────┐
  │         設定項目         │ 推奨値 │                理由                 │
  ├──────────────────────────┼────────┼─────────────────────────────────────┤
  │ Require a pull request   │ ✅     │ featureブランチは必ずPR経由でマージ │

  ---
  サポートブランチへの命名規則の強制

  Settings → Branches → Add branch ruleset でパターンを指定することで、命名規則
  に沿ったブランチしか作成できないよう制限できます。

  # 許可するブランチ名パターン（Ruleset で設定）
  feature/*
  release/*
  hotfix/*

  逆に、上記パターン以外のブランチ名を拒否する設定も可能です。

  ---
  設定の全体像まとめ

  main        ─── 最厳格：PR必須 / 承認必須 / CI必須 / 直接push完全禁止
  develop     ─── 中程度：PR必須 / 承認必須 / CI必須 / 直接push禁止
  feature/*   ─── 保護なし（開発者が自由に作業）
  release/*   ─── 保護なし（リリース担当者が作業）
  hotfix/*    ─── 保護なし（緊急対応者が作業）

  ---
  補足：Require linear history を有効にしてはいけない理由

  git-flowは --no-ff によるマージコミットを戦略の中核に置いています。Require 
  linear 
  history（squash/rebaseのみ許可）を有効にすると、マージコミットが作れなくなり、

  ---
  Q1. 設定したパターン以外のブランチは一切作成できなくなるか？

  設定の仕方によります。

  GitHubのブランチ設定には2種類あり、挙動が異なります。

  ┌────────────────┬────────────────────────────────────────────────────────┐
  │      機能      │                          動作                          │
  ├────────────────┼────────────────────────────────────────────────────────┤
  │ Branch         │ 指定したパターンのブランチを保護する設定。他のブランチ │
  │ protection rul │ 名の作成は制限しない                                   │
  │ es（従来型）   │                                                        │
  ├────────────────┼────────────────────────────────────────────────────────┤
  │ Rulesets（新し │ 「Restrict creations」ルールと組み合わせることで、指定 │
  │ い機能）       │ パターン以外のブランチ作成を拒否できる                 │
  └────────────────┴────────────────────────────────────────────────────────┘

  命名規則を強制したい場合は Rulesets を使い、以下のように設定します。

  【Rulesetの設定例】
  対象: 全ブランチ（All branches）
  ルール: Restrict creations（ブランチ作成を制限）
  除外（Bypass）パターン:
    - main
    - develop
    - feature/*
    - release/*
    - hotfix/*

  この設定により、上記以外の名前でブランチを作成しようとするとGitHubに拒否されま
  す。ただし、チームの規模や運用によっては技術的強制より運用規約として周知するに
  とどめるケースも多く、厳格な強制が必ずしも必要とは限りません。

  ---
  Q2. この規則はリモートのみ？ローカルも？

  リモート（GitHub）のみに適用されます。ローカルには影響しません。

  ローカル: どんな名前でもブランチを作成・コミット可能
                  ↓ git push
  リモート(GitHub): Rulesetによりブランチ名を検証 → 規則違反なら拒否

  つまり、ローカルで git checkout -b test-branch と作業しても問題ありませんが、
  それをGitHubにpushしようとした時点でルールが適用されて拒否されます。

  ---
  1. Git hooks によるローカル強制（commitlint + husky）

  commitlint はコミットメッセージをConventional
  Commits規約に照らして検証するツールです。husky はGit
  hooksをチームで共有しやすくするツールで、この2つを組み合わせるのが定番です。

  # セットアップ例（Node.js プロジェクトの場合）
  npm install --save-dev @commitlint/cli @commitlint/config-conventional husky

  # huskyの初期化
  npx husky init

  # commit-msg フックに commitlint を登録
  echo "npx --no -- commitlint --edit \$1" > .husky/commit-msg

  // commitlint.config.js
  export default {
    extends: ['@commitlint/config-conventional'],
  };

  この設定により、規約外のコミットメッセージはローカルで即座に弾かれます。

  $ git commit -m "ログイン機能を追加"
  ⧗ input: ログイン機能を追加
  ✖ subject may not be empty [subject-empty]
  ✖ type may not be empty [type-empty]
  ✖ found 2 problems, 0 warnings

  注意点: Git hooksはローカルの .git/
  に存在するためリポジトリ管理外です。huskyを使うことで package.json
  のインストール時に自動セットアップされ、チームで統一できます。

  ---
  2. GitHub Actions によるCI強制

  Git hooksはあくまでローカルの仕組みなので、huskyをインストールしていない開発者
  には効きません。CIで二重に検証するのがより確実です。

  接頭辞の基本検証

  # .github/workflows/commitlint.yml
  name: Commit Lint

  on:
    push:
      branches:
        - 'feature/**'
        - 'release/**'
        - 'hotfix/**'

  jobs:
    commitlint:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
          with:
            fetch-depth: 0
        - uses: actions/setup-node@v4
        - run: npm ci
        - run: npx commitlint --from origin/develop --to HEAD

  release/* / hotfix/* での feat: を検出してエラー

  ご指摘の通り、これをCIで検知することも可能です。

  # .github/workflows/branch-commit-rule.yml
  name: Branch Commit Rule Check

  on:
    push:
      branches:
        - 'release/**'
        - 'hotfix/**'

  jobs:
    check-no-feat:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
          with:
            fetch-depth: 0
        - name: feat:コミットが含まれていないか確認
          run: |
            BRANCH=${GITHUB_REF#refs/heads/}
            COMMITS=$(git log origin/main..HEAD --pretty=format:"%s")
            if echo "$COMMITS" | grep -q "^feat:"; then
              echo "ERROR: $BRANCH ブランチに feat: コミットが含まれています。"
              echo "release/hotfix ブランチでの新機能追加は git-flow 
  のルール違反です。"
              exit 1
            fi

  このワークフローにより、release/* や hotfix/* ブランチで feat: を含むコミット
  をpushした時点でCIが失敗し、PRのステータスチェックもブロックされます（Branch
  protection rulesの「Require status
  checks」と組み合わせることで効果を発揮します）。
